Orders of Growth: RECAP

목표:

  • 입력이 매우 클 때 프로그램의 효율성을 측정

  • 입력 크기가 증가함에 따라 프로그램의 실행 시간이 늘어나는 것을 표현

  • 향상의 상한선(upper bound)을 최대한 엄격하게 정함

  • 정확할 필요는 없음 - it’s "order of" not "exact"

  • 런타임에서 가장 큰 요소를 살펴봄 (프로그램의 어떤 부분이 실행에 가장 오랜 시간을 소요하는가?)

  • 따라서 일반적으로 최악의 경우 입력 크기의 함수로서 향상의 상한선을 정함

Complexity Classes(복잡성 분류): RECAP

  • \(O(1)\)은 일정한 실행 시간을 나타냄

  • \(O(log n)\)은 로그 실행 시간을 나타냄

  • \(O(n)\)은 선형 실행 시간을 나타냄

  • \(O(n log n)\)은 로그 선형 실행 시간을 나타냄

  • \(O({n}^{c})\)는 다항식 실행 시간을 나타냄(c는 상수)

  • \(O({c}^{n})\)은 지수 실행 시간을 나타냅니다. (c는 입력 크기에 따라 거듭제곱되는 상수)

Constant Complexity

  • 입력과 무관한 복잡성

  • 이 클래스에는 흥미로운 알고리즘이 거의 없지만, 이 클래스에 맞는 부분이 있는 경우가 많음 반* 복 또는 재귀 호출이 있을 수 있으나 입력 크기와 관계 없이 반복 또는 호출 횟수가 있는 경우에 해당

Logarithmic Complexity(로그 복잡도)

  • 입력 중 하나의 크기에 대한 로그만큼 복잡성이 증가

  • 예:

    • 이분 검색(bisection search)

    • list의 이진 검색

Bisection Search(이분 검색)

  • 특정 요소가 list에 있는지 알고 싶다고 가정

  • 지난번에는 list를 "아래로 내려가면서" 각 요소를 확인할 수 있다는 것을 확인

  • 복잡성은 list의 크기에 따라 선형적

  • list가 가장 작은 것 부터 가장 큰 것 순으로 정렬되어 있다는 것을 알고 있다고 가정

    • 순차 검색이 복잡성 면에서 선형일 것이라는 것을 확인

    • 더 나은 방법?

Bisection Search - 과정

  1. list를 반으로 나누는 인덱스를 검색

  2. L[i] == e인지 확인

  3. 아니면 L[i]이 ‘e’보다 큰지 또는 작은지 확인

  4. 답에 따라, e를 검색하기위해 왼쪽 또는 오른쪽의 절반을 검색

divide-and-conquer(분할과 정복) 알고리즘의 새로운 버전

  • 더 작은 버전의 문제(smaller list)와 몇 가지 간단한 작업으로 분류

  • 작은 버전의 해답은 원래 문제의 해답이 됨

Bisection Serarch - 복잡도 분석

  • 아래에 해당하면 검색이 종료됨

    • 1 = N / 2i

    • 그러므로 i = log n

  • 재귀 복잡도는

    • O(log n) – 여기서 n은 len(L)

첫번째 이분 검색 구현

def bisect_search1(L, e):
    if L == []:          # constant O(1)
        return False
    elif len(L) == 1:    # constant O(1)
        return L[0] == e
    else:
        half = len(L) // 2 # constant O(1)
        if L[half] > e:
            return bisect_search1(L[:half], e) # NOT constant
        else:
            return bisect_search1(L[half:], e) # NOT constant

  • L[half:] - Not constant, copies list

첫 번째 이분 검색 방법 복잡도

bisect_search1

  • O(log n) 이분 검색은

    • 각각의 재귀 호출에서, 검색할 범위는 반으로 절단됨

    • 원래의 크기가 n이면, n/(2^k) = 1일 때 또는 k = log n일 때 크기가 1까지 줄어듦

  • 각 이분 검색에서 각 복사된 list에 대해 O(n)

    • 이것은 각각의 호출에 대한 비용으로, 각 단계의 재귀는 이를 행함

  • O(log n) * O(n) → O(n log n)

  • 정말 신중하다면, list의 길이 만큼이 복사되어 각 재귀에서 사용된다는 것을 기억할 것

    • 복사 비용은 O(n)이며 이는 재귀 비용으로 인한 log N 비용

다른 방법의 이분 검색

  • 각 단계에서 문제의 크기를 2배로 줄이는 것은 같음

  • 검색할 목록의 낮은 부분과 높은 부분만 추적

  • list 복사를 회피

  • 재귀 복잡성은

    • O(log n) - 여기서 n은 n이 len(L)

두번째 이분 검색 구현

def bisect_search2(L, e):
    def bisect_search_helper(L, e, low, high):
        if high == low:
            return L[low] == e
        mid = (low + high)//2
        if L[mid] == e:
            return True
        elif L[mid] > e:
            if low == mid: #nothing left to search
               return False
            else:
                # constant other than recursive call
                return bisect_search_helper(L, e, low, mid - 1)
        else:
            # constant other than recursive call
            return bisect_search_helper(L, e, mid + 1, high)
    if len(L) == 0:
        return False
    else:
        return bisect_search_helper(L, e, 0, len(L) - 1)

두 번째 이분 검색 방법 복잡도

bisect_search2 와 helper

  • O(log n) 이분법 검색 호출

    • 각 재귀 호출 마다 검색 범위 크기가 절반으로 줄어듬

    • 원래 범위의 크기가 n 이면 최악의 경우 n(2k) 일 때, 또는 k = log n 일때 크기 1의 범위까지 내려감

  • list과 index를 파라미터로 전달

  • list는 복사되지 않고 포인터로 다시 전달됨

  • 따라서 O(1) 은 각 재귀 호출에 대해 작업

  • O(log n) * O(1) → O(log n)

Logarithmic Complexity(로그 복잡도)

def intToStr(i):
    digits = '0123456789'
    if i == 0:
        return '0'
    result = ''
    while i > 0:
        result = digits[i % 10] + result
        i = i // 10
    return result

  • 함수 호출이 없는 반복만 살펴보면 됨

  • while 반복 내에서 일정한 단계 수

  • 루프를 몇 번이나 통과할까?

    • i를 10으로 몇 번 나눌 수 있나?

    • O(log(i))

Linear Complexity(선형 복잡도)

  • 이미 살펴 보았음

    • 요소가 존재하는지 확인하기 위해 순서대로 list 검색

    • Iterative loops

Iterative Factorial의 O(n)

  • 복잡도는 iterative call의 숫자에 따름

    def fact_iter(n):
    prod = 1
    for i in range(1, n+1):
        prod *= I
    return prod

  • 전체적으로 O(n) – n번의 round loop, 각각 상수 비용

Recursive Factorial의 O(n)

def fact_recur(n):
    """ n >= 0 이라고 가정 """
    if n <= 1:
        return 1
    else:
        return n * fact_recur(n)

  • factorial을 다시 불러(recursive) 계산

  • 시간을 측정하면 기능으로 인해 iterative 버전보다 조금 느리게 실행됨

  • 함수 호출 수가 n에서 선형이고 지속적으로 호출하므로 O(n)

  • iterative와 recursive factorial은 같은 증가 기준(same order of growth)을 가짐

Log-Linear Complexity (로그-선형 복잡도)

  • 많은 실용적인 알고리즘은 로그-선형적임

  • 매우 일반적으로 사용되는 로그-선형 알고리즘은 병합 정렬

  • 여기서는 설명하지 않음

Polynomial Complexity(다항식 복잡도)

  • 가장 일반적인 다항식 알고리즘은 2차 알고리즘으로, 입력 크기의 제곱에 따라 복잡성이 증가함

  • 중첩 루프나 재귀 함수 호출이 있을 때 일반적으로 발생함

Exponential Complexity(기하급수적 복잡도)

  • 각 문제에 대해 둘 이상의 재귀 호출이 있는 재귀 함수

    • 하노이 타워

  • 많은 중요한 문제는 본질적으로 기하 급수적

    • 비용이 많이 듦

    • 보다 신속하게 합리적인 답을 제공할 수 있는 대략적인 솔루션을 고려하도록 유도

Complexity of Towers of Hanoi(하노이 타워 복잡도)

  • \({t}_{n}\)은 크기 n의 타워를 해결할 시간을 나타냄

    \[\begin{align*} tn &= 2t_{n-1} + 1\\ &= 2(2t_{n-1} + 1) + 1\\ &= 4t_{n-1} + 2 + 1\\ &= 4(2t_{n-3} + 1) + 2 + 1\\ &= 8t_{n-3} + 4 + 2 + 1\\ &= {2}^{k}t_{n-k} + 2^{k-1} + … + 4 + 2 + 1\\ &= 2^{n-1} + 2^{n-2} + … + 4 + 2 + 1\\ &= 2^{n-1} \end{align*}\]

  • Order of Growth는 \(O(2^{n})\)

Geometric growth(기하학적 성장)

\[\begin{align*} a &= {2}^{n-1} + … + 2 + 1\\ 2a &= {2}^{n} + {2}^{n-1} + … + 2\\ a &= {2}^{n} - 1 \end{align*}\]

Power Set - Concept

  • 1에서 n까지 정수의 멱집합을 구하고자 한다

  • 1부터 n-1까지 정수의 거듭제곱 집합을 생성할 수 있다고 가정

  • 그런 다음 해당 부분 집합 모두는 더 큰 멱집합에 속합니다. N이 더 큰 멱집합에 해당하는 모든 부분 집합(n을 포함하지 않음 선택) 그리고 그들 각각에 또한 추가됨 (n을 포함하도록 선택)

def genSubsets(L):
    res = []
    if len(L) == 0:
        return [[]]     # list of empy list
    smaller = genSubsets(L[:-1])    # all subsets without last element
    extra = L[-1:]      # create a list of just last element
    new = []
    for small in smaller:
        new.append(small + extra)   # for all smaller solutions, and one with last element
    return smaller + new            # combine those with last element and those without
